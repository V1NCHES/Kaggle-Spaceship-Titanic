# Spaceship Titanic: Пошаговое решение ML задачи

Этот репозиторий содержит решение задачи **Spaceship Titanic** с платформы Kaggle.

## 🚀 Описание задачи

**Spaceship Titanic** — это соревнование по машинному обучению, вдохновленное классической задачей Титаника, но в сеттинге будущего.

**Сюжет:** Космический корабль "Титаник" столкнулся с пространственной аномалией. Почти половина пассажиров была перенесена в другое измерение. Наша задача — предсказать, какие пассажиры были перенесены (`Transported`), на основе данных об их полете, возрасте, планете и тратах на борту.

**Источник:** [Kaggle Spaceship Titanic Competition](https://www.kaggle.com/competitions/spaceship-titanic)

---

## 🛠 Что было сделано (Продвинутое решение)

В обновленной версии решения (`Spaceship_Titanic_Advanced_Solution.ipynb`) реализованы следующие улучшения:

1.  **Продвинутый Feature Engineering**:
    *   **Family & Group Imputation**: Восстановление пропущенных планет (`HomePlanet`) на основе данных о группе (`Group`) и фамилии (`Surname`). Семьи обычно путешествуют вместе.
    *   **CryoSleep Logic**: Если у пассажира есть траты (`TotalSpending > 0`), он гарантированно не был в криосне.
    *   **Cabin Regions**: Каюты разбиты на регионы по 300 номеров для улавливания локальных зависимостей.
    *   **Log Transformations**: Логарифмирование трат (`TotalSpending`) для нормализации распределения с "тяжелыми хвостами".
2.  **Ансамбль моделей (Ensemble Learning)**:
    *   Объединение трех мощных алгоритмов градиентного бустинга: **CatBoost**, **XGBoost** и **LightGBM**.
    *   Использование `VotingClassifier` с мягким голосованием (`soft voting`) для повышения стабильности предсказаний.
3.  **Глубокая валидация**:
    *   Оценка моделей не только по Accuracy, но и по метрике **ROC-AUC** для более точного понимания качества классификации.

## Результаты и Оптимизация

### Оптимизация моделей (Апрель 2026)
В ходе анализа была выявлена и устранена критическая ошибка в предобработке данных:
- **Исправление кодирования**: Ранее `LabelEncoder` применялся отдельно к Train и Test, что приводило к несоответствию признаков. Теперь используется единая логика кодирования, гарантирующая идентичность значений.
- **Улучшенная обработка пропусков**: Реализовано «умное» заполнение `HomePlanet` и `CryoSleep` на основе данных о группах и тратах пассажиров.
- **Новые признаки**: Добавлены `FamilySize` (размер семьи по фамилии) и уточненный `CabinRegion`.
- **Ансамбль моделей**: Итоговая точность на кросс-валидации (Accuracy) достигла **~0.8140**, ROC-AUC **~0.9025**.

### Модели и Метрики
Для выбора лучшей модели теперь используется **Accuracy** как основная метрика (согласно требованиям Kaggle), а ROC-AUC, Precision, Recall и F1 используются для глубокой диагностики:
| Алгоритм | Accuracy | ROC-AUC | F1-Score |
| :--- | :--- | :--- | :--- |
| CatBoost | ~0.8120 | ~0.8980 | ~0.8110 |
| XGBoost | ~0.8050 | ~0.8910 | ~0.8030 |
| LightGBM | ~0.8080 | ~0.8950 | ~0.8060 |
| **Ensemble (Voting)** | **0.8140** | **0.9025** | **0.8125** |

### Ключевые выводы
- Использование стандартного градиентного бустинга (CatBoost, XGBoost, LightGBM) позволяет достичь базового уровня Accuracy около 0.80-0.81.
- Результат можно существенно улучшить за счет **ансамбля из 3 моделей** (Voting Classifier) и **продвинутой обработки пропусков** (логическое восстановление HomePlanet, CryoSleep и др.).
- Хотя Accuracy является целевой метрикой, отслеживание **ROC-AUC** помогает лучше понять разделяющую способность модели и выбрать более стабильные гиперпараметры.
